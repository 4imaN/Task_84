# LedgerRead Design

## Technical Architecture

LedgerRead is a containerized npm-workspaces monorepo with two runtime services: PostgreSQL and a unified NestJS application container that serves both the REST/GraphQL backend and the built React frontend. PostgreSQL is the system of record for users, sessions, titles, chapters, carts, inventory, settlements, moderation, attendance, and audit chains. The React client consumes GraphQL for high-read catalog and community workloads while REST is reserved for transactional flows such as authentication, checkout, attendance, moderation actions, reconciliation imports, and reading-profile synchronization. GraphQL playground and introspection are disabled outside development.

## Trust Boundaries

1. Browser: stores encrypted title payloads and reading profiles locally in IndexedDB, but authentication stays in an `httpOnly` session cookie instead of browser storage.
   - The offline cache uses a per-user non-extractable AES key generated by Web Crypto and stored in IndexedDB, so title/profile encryption is not derived from predictable clear-text values in browser storage.
   - Persisted browser-storage keys are also obfuscated and legacy clear-text keys are migrated forward, so usernames and title ids are no longer readable from `localStorage` or IndexedDB key names.
2. API: enforces authentication, authorization, anti-spam, audit creation, and encryption-at-rest policies.
3. Database: stores only encrypted sensitive fields and immutable audit records.
4. Local storage volume: holds attendance evidence files with checksum metadata.

## Route Partitions

1. `/app/*`: Customer workspace with reader and community experiences.
2. `/pos/*`: Clerk workspace for checkout and attendance.
3. `/mod/*`: Moderator workspace for report queue processing.
4. `/admin/*`: Shared manager and inventory workspace for oversight, inventory actions, and the shared reconciliation/audit review surface.
5. `/finance/*`: Finance-only workspace for the same reconciliation review surface and audit review.

Clerk POS flow includes a dedicated inventory discovery endpoint (`GET /pos/search?q=...`) so staff can find SKUs quickly while constructing carts before running `review-total` and `checkout`.

Each workspace has its own login page and layout shell. The API independently revalidates every request so direct URL entry or forged requests still receive `403 Forbidden`.
The admin and finance workspaces also apply page-level route guards and navigation filtering so:
- `MANAGER` can reach `/admin/overview`, `/admin/finance`, `/admin/inventory`, and `/admin/audits`
- `INVENTORY_MANAGER` can reach `/admin/overview`, `/admin/finance`, `/admin/inventory`, and `/admin/audits`
- `FINANCE` stays isolated to `/finance/settlements` and `/finance/audits`
- `/admin/finance` and `/finance/settlements` reuse the same reconciliation screen so finance and inventory users both see settlement status, discrepancy flags, and direct audit-trail access in one review context
- Manifest imports remain available only inside the manager and inventory-admin workspace, so finance stays read-only for settlement intake while still sharing the same discrepancy and audit review context.

## Domain Data Models

Core entities:

- `users`: identity, role, password hash, lockout state, encrypted identifier fields
  - login usernames are stored as `username_cipher` plus a deterministic `username_lookup_hash`, so authentication can look users up without persisting plaintext usernames at rest
- `sessions`: opaque session tokens with idle timeout tracking
- `titles`, `chapters`, `authors`, `series`: catalog and reader payloads
- `reading_profiles`: persisted preferences and LAN-sync timestamps
- `favorites`, `author_subscriptions`, `series_subscriptions`, `ratings`
  - the customer community query hydrates viewer-specific favorite, author-follow, and series-follow state directly from these tables so the controls do not reset between title switches.
- `comments`, `reports`, `moderation_actions`, `user_blocks`, `user_mutes`
  - replies are validated server-side so `parent_comment_id` must belong to the same title thread as the new comment
  - report category and notes are validated as non-empty metadata at the API boundary, including whitespace-only tampering attempts
- `inventory_items`, `bundle_links`, `carts`, `cart_items`, `orders`, `order_items`
  - carts persist `review_signature`, `reviewed_at`, `reviewed_total_cents`, and a review snapshot so checkout is blocked unless a fresh review exists.
- `supplier_statements`, `supplier_statement_lines`, `supplier_invoices`, `supplier_invoice_lines`, `inventory_receipts`, `payment_plans`, `reconciliation_discrepancies`
  - imports reconcile supplier statement quantities and amounts against invoice quantities and amounts, allocate freight and surcharges into landed cost, and update moving-average inventory valuation from purchase cost instead of retail price.
- `attendance_records`, `risk_alerts`, `rule_versions`
- `audit_logs`: immutable chain-linked mutation log with trace IDs
- `recommendation_snapshots`, `recommendation_traces`: precomputed Top-N/similar-item results plus persisted trace rows for snapshot hits, cache hits, and fallbacks.

## Edge Cases Planned

1. Stale session access after 30 idle minutes returns `401` and requires re-authentication.
   - The React client revalidates stored sessions on startup and clears client state on centralized `401`/`403` responses before redirecting back to the correct workspace login page.
2. Customer attempts to access `/pos/*`, `/mod/*`, or `/admin/*` are blocked in both UI and API.
3. LAN profile sync detects stale timestamps and can return `409 Conflict`.
4. Comment duplicate detection rejects same-body submissions within 60 seconds.
5. Attendance missing clock-out beyond 12 hours raises risk alerts.
   - The backend now evaluates this rule in a scheduled background pass every 30 minutes, so overdue alerts do not depend on the affected user opening the attendance UI.
6. Evidence checksum mismatch raises risk alerts and is recorded in the audit chain.
   - Evidence uploads no longer trust browser MIME alone. The API validates PNG/JPEG/WebP signatures, generates server-side filenames, and keeps resolved paths inside the configured evidence root.
7. Checkout revalidation prevents stale stock or tampered price submissions, and checkout is rejected if the cart changed after the last successful review.
   - Any cart line add, exact-quantity edit, or deletion clears the stored review snapshot so clerks must rerun `review-total`.
8. Recommendation resolver falls back to best-sellers after a 150 ms timeout and records trace rows for every response path, including cache hits.
9. Cross-user reading-profile access is intentionally absent; only `/profiles/me` is exposed.
10. Supplier imports are transactional, so missing SKUs or other failures roll back statements, invoices, receipts, and payment-plan writes together.
11. Reader language switching is instant because the GraphQL title payload includes both Simplified and Traditional chapter bodies, allowing the active session to swap rendered text without refetching.
12. Moderation restore is surfaced through a resolved-history queue, while suspend derives the reported account on the server from report/comment context.
13. Browser refreshes do not reuse stale in-memory auth state. The web app rehydrates from `/auth/session`, clears state on centralized `401`/`403`, and returns the user to the correct login entry point.
14. Local attendance evidence UX computes SHA-256 in the browser, shows match/mismatch feedback before submit, and disables submission on mismatched checksums.
15. Important write flows use one shared frontend mutation pattern with loading states, duplicate-click prevention, normalized toasts, and safe recovery for community actions, POS, attendance, moderation, finance imports, and reading-profile portability.
16. File-based profile import now follows the same newest-timestamp rule as LAN sync:
   - local state keeps the current profile when the imported file is older
   - newer imported files are resolved against server state through `/profiles/me/sync`
   - stale imported files can no longer overwrite a newer server profile
17. Reader and profile surfaces now terminate cleanly:
   - `ReaderPage` resolves into loading, encrypted-cache fallback, retryable error, or success
   - `ProfilePortabilityPage` resolves into loading, success with optional recovery banner, or retryable error

- One reviewer account per major role
- Seeded credentials:
  - `reader.ada` / `Reader!2026`
  - `reader.mei` / `Reader!2026`
  - `clerk.emma` / `Clerk!2026`
  - `mod.noah` / `Moderator!2026`
  - `manager.li` / `Manager!2026`
  - `finance.zoe` / `Finance!2026`
  - `inventory.ivan` / `Inventory!2026`
- Sample titles with chapters
- Inventory SKUs with complementary bundle links
- Moderation-sensitive words
- Baseline governance rules
- Seeded payment-plan examples plus deterministic reconciliation tables that are reset before each reseed

## Security Notes

1. Passwords must be at least 10 characters and include one number and one symbol.
2. Hashing uses salted Argon2.
3. Sensitive fields such as payment notes, external identifiers, and login usernames are AES-GCM encrypted before persistence.
   - authentication resolves accounts through a keyed `username_lookup_hash` and only decrypts the username when the application needs to display it back to an authenticated user
   - Direct seed/config startup now fails if `APP_ENCRYPTION_KEY` is missing or uses the banned demo secret.
   - The container startup wrapper and packaged test wrappers share a generated runtime key file under `.ledgerread-runtime/app_encryption_key`, so reviewer flows stay zero-manual-setup without rekeying encrypted rows across reruns.
4. Trace IDs are attached to auditable mutations and recommendation responses and stored in PostgreSQL.
5. Tests suppress tokens and passwords from command output, and browser storage no longer holds bearer tokens or raw offline cache secrets.
